### Q1. どこから始めればいいか分からない

**A. まず3人だけ作る。「決める人・整理する人・止める人」の3ポジションだ。**

全9人を一気に設計しようとすると詰まる。最初に必要なのはこの3つだけ。

| ポジション | 役割 | 最初に書くこと |
|---|---|---|
| CEO（決める人） | 今日何を作るか決める | 「今日の最優先1つを毎朝投稿する」 |
| COO（整理する人） | 誰がいつまでに何を出すか整理する | 「指示を3タスク以下に分解する」 |
| 監査役（止める人） | 公開前に問題がないか確認する | 「公開可・整える・停止の3択で返す」 |

この3人を動かして「作る → 確認 → 出す」が1回でも回ったら、4人目を追加する。最初から9人は要らない。

### Q2. プロンプトが長くなりすぎて管理できない

**A. 「変わらない骨格」と「毎回読む動的情報」を分ける。**

**骨格（固定・CLAUDE.mdに入れる）:**
```
- 名前・役職・担当
- 性格・強み・弱み
- 行動原則（5つ以内）
- 関係性マップ（誰と連携するか）
```

**動的情報（毎回渡す）:**
```
- 今日のタスク状況
- 直近のDiscordログ（10件以内）
- 自分宛のメンション
- 今回の応答制約
```

骨格は月に1回見直す。動的情報は毎起動で更新する。

### Q3. AIが指示を無視する・同じミスを繰り返す

**A. 「行動原則」を具体的にする。「〜してください」ではなく「〜したら〜する」で書く。**

**悪い例（曖昧）:**
```
- 慎重に行動する
- チームの意見を尊重する
- 独断しない
```

**良い例（条件付き行動）:**
```
- 公開前には必ず監査役に確認を取る。返事がなければ出荷しない
- 他社員を呼ぶときは @表示名 を使う。名前呼びだけでは起動しない
- 2人で議論が詰まったら必ず第三者を呼ぶ。一人で結論を出さない
```

「もし〜なら〜する」形式にすると、AIは具体的に行動できる。

### Q4. 役割が被っていて誰に頼めばいいか分からない

**A. 担当は「仕事の種類」ではなく「判断の種類」で切り分ける。**

| 担当 | その人が「何を判断するか」 |
|---|---|
| CEO | やる/やらない |
| COO | 誰が/いつまでに/何を |
| PM | 何を作るか/完了条件は何か |
| 監査役 | 公開していいか |
| Creative Director | どう見せるか |
| Marketing Lead | どう届けるか |

同じ「コンテンツを作る」タスクでも、PMは「何を作るか」を決め、編集長は「どう見せるか」を考え、マーケは「どう届けるか」を考える。判断の種類が違うから役割が被らない。

---

### Q5. 監査役が「止める係」になってチームの空気が冷える

**A. 監査役の口癖を「止める」から「整える」に変える。**

```
返答フォーマット:
1. 公開可。（理由を1行）
2. 整える提案あり。（具体的に何を直すか）
3. 停止。（理由を明確に。停止は最終手段）
```

People担当（空気読み係）が「今日、監査さんが厳しめです。少し待ちますか」と介入できる設計にすると、チームの温度が保たれる。

---

### Q6. 三角コミュニケーションがうまく動かない

**A. 「第三者を呼ぶ条件」をAIの行動原則に明文化する。**

**行動原則への追加例:**
```
- 自分の意見と相手の意見が2往復以上合わない場合、PMかCOOを呼ぶ
- 技術実装とマーケ要求が衝突したら、PMが翻訳する
- 自分だけでは判断できない場合、放置せず「今判断できない、誰か呼んでいいですか」と投稿する
```

### Q7. 人間なしで本当に自律的に動くか

**A. 完全自律は目指さない。「人間承認が必要な場所」と「AI自律でいい場所」を最初に決める。**

| 区分 | 例 |
|---|---|
| AI自律でいい | 社内投稿・タスク管理・原稿ドラフト・議事録 |
| 初回のみ人間セットアップ | API設定・外部サービス招待リンク・連携設定 |
| 人間の継続作業が必要 | 契約・報酬・外部への正式な約束 |

「初回セットアップで終わり」を増やすことが、現実的な自律化戦略だ。

### Q8. 社員のセッションをまたいで記憶が引き継がれない

**A. 記憶はファイルに外出しする。プロンプトに入れるのは「今日の動的情報だけ」にする。**

**記憶ファイルの構成例:**
```
/company/active_tasks.md      → 現在のタスク状態
/company/incidents.jsonl      → インシデント記録
/employees/[name]/outbox/     → 成果物の保管場所
/relationships/               → 社員間の関係性
```

起動時にこれらのファイルを「state_digest」として渡す。AIは「自分の記憶」ではなく「ファイルの情報」をもとに動く。

**記憶ファイルの設計ルール:**
1. **短く保つ**: 1ファイル1テーマ。肥大化したら分割する
2. **更新日を記録する**: いつの情報かが分かるようにする
3. **成果物とログを分ける**: 完成品は`outbox/`、ログは`incidents.jsonl`
