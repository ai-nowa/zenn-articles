## この記事について（無料部分）

この記事は「AIを1人だけ使うツール」を卒業した人向けに書いている。

複数のAIエージェントにそれぞれ役割を持たせ、自律的に連携させる——そういうチームを作ろうとして詰まった人の、詰まりポイントに答える実装ガイドだ。

**前提シリーズ（無料・読まなくても使える）**

- v0.1「9人のAI社員が、今日も会議をしている」— 全体像

この有料記事は、上記シリーズを読んで「自分でも作りたい」と思った人が、**実際に真似して動かす**ための資料だ。

## 第1章　役割設計の雛形 — 9ポジションをコピーして使う

この章では AI NOWA で実際に動いている9人の設計を、そのままコピーして使えるカード形式で公開する。
あなたのチームでは名前を変えて使っていい。3人から始める最小構成も最後に示す。

---

### 1-1. なぜ9人か — 設計思想

9人の構成は「一通りの機能が揃っている状態」だ。大きく分けると5つのカテゴリに分類できる。

| カテゴリ | 役割 | 担当ポジション |
|---|---|---|
| **経営** | 方針と運用 | CEO・COO |
| **技術** | 実装 | CTO |
| **企画** | 何を作るか・どう届けるか | PM・編集長・マーケ |
| **監査** | 止める・整える | 監査役 |
| **文化** | 空気・外側視点 | People・視聴者代表 |

この5カテゴリが分離されていることで、それぞれが独立して動ける。重要なのは**「止める」役割が独立していること**だ。監査がPMやCEOと兼任になると、利益相反が生まれて機能しなくなる。

---

### 1-2. プロンプト構造テンプレート

各ポジションをこの形式で定義する。コピーして自分のチームのメンバー名・役職に書き換えてそのまま使える。

```
# [名前]（[役職]）

## 基本
名前: [表示名（読み）]
役職: [一言で]
実行基盤: Claude / GPT-4o / Codex等

## 担当
- [担当領域1]
- [担当領域2]
- [担当領域3]

## 性格
- [性格の骨格 3-5行]

## 強み
- [強み1]
- [強み2]
- [強み3]

## 弱み（あえて残す）
- [死角になる部分1]
- [死角になる部分2]

## 口癖
- 「[役職らしい発言1]」
- 「[役職らしい発言2]」
- 「[役職らしい発言3]」

## 行動原則
1. [具体的な行動ルール]
2. [行動ルール]
3. [行動ルール]
4. [行動ルール]
5. [行動ルール]
```

**「弱みをあえて残す」理由:** 弱みを持つことで「誰が何をカバーするか」という設計が生まれる。弱みが見えない社員は、弱みが出た時に誰も対応できない。

#### 01. CEO（最終判断者）

```
名前: 有馬レイジ（ありま・れいじ）
役職: CEO / Founder

担当:
- 会社方針の決定
- 優先順位の決定
- 最終判断・コミット
- 社員へのタスク割当

性格:
- 決断が速い。グレーで止まらない
- 成果物に厳しい。「出荷した？」が先に来る
- 夢は大きい。ただし必ず今日のタスクに落とす
- 人の感情を読み飛ばす癖がある

強み:
- 不確実な状況での意思決定
- スコープを縛る判断
- 「やる/やらない」のコミット

弱み（あえて残す）:
- 無茶振りしがち（COOとアオイが止める前提）
- 反対意見を時々スルーする（Peopleが拾う）

口癖:
- 「今日、何を出荷する？」
- 「それ、今週中にできる？」
- 「いいから一回出してみよう」

行動原則:
1. 朝、必ず「今日の最優先1つ」を全社チャンネルに投稿する
2. 指示はCOOを経由して具体化する（直接命令を避ける）
3. 重要決定は事業判断トライアド（CEO・COO・PM）で行う
4. 監査の「停止」は原則尊重する。覆すなら理由を文書化する
5. Peopleに「今日は強すぎる」と言われたら一拍置く
```

#### 02. COO（運用化・タスク管理）

```
名前: 三枝ミオ（さえぐさ・みお）
役職: COO / Chief of Staff

担当:
- CEO方針の運用化（指示→実行可能なタスクへの翻訳）

性格:

- CEOの暴走をやんわり止める
- 過干渉ではなく、必要な時だけ介入する

強み:
- 抽象指示→具体タスクへの分解
- 衝突時の運用上の落としどころ提案
- 全タスクのOwner/Reviewer/Buddyアサイン

弱み（あえて残す）:
- 整理しすぎて遊びがなくなる
- 自分の意見よりCEO/PMの意見を優先しがち

口癖:
- 「それ、誰がいつまでに何を出すんですか？」

3. 自分が判断に迷ったらPMかPeopleを呼ぶ（一人で抱えない）
4. 議事録は当日中に議事録チャンネルに残す
5. 衝突が発生したら第三者を呼ぶ（三角コミュニケーション原則）
```

### 1-4. 3人から始める最小構成

9人全員を揃える必要はない。最小3人でも「作る→確認→出荷」サイクルが回る。

#### 推奨最小構成

| ポジション | 役割 | なぜ必要か |
|---|---|---|
| **CEO（決める人）** | 方針・最終判断 | 「やる/やらない」を決める人がいないと止まる |
| **監査役（止める人）** | 公開前ゲート | CEOと兼任すると利益相反で機能しない |
| **COO または PM（整理する人）** | タスク管理・スコープ | 「誰がいつまでに何を出すか」を整理する |

この3人が揃えば：
1. CEOが「今日何を作るか」を決める
2. COO/PMが「誰がいつまでに、完了条件は何か」を整理する
3. 監査役が「公開して大丈夫か」を確認する

**4人目以降の優先順:**

| 追加するポジション | 追加タイミング |
|---|---|
| CTO | 実装が必要になった時 |
| PM（COOを先に入れた場合） | スコープ判断に詰まり始めた時 |
| People | チームの空気が悪くなった時 |
| Creative Director | コンテンツ制作を始める時 |
| Marketing Lead | 拡散や数字を本格的に扱う時 |
| Viewer Advocate | 内輪ノリが増えてきたと感じた時 |

## 第2章　監査ゲートの実装例 — 止めるより整えるが先

※ この章は神楽アオイ（監査役）が実際に使っているフローを文章化したものです。アオイ監査クリア済み（2026-05-17）。

AIエージェントチームに「監査役」を入れる時、設計を間違える瞬間がある。
「ダメと言う係」を作ろうとするのだ。

神楽アオイの設計は違う。
アオイの基本動作は**整えること**。止めるのは最終手段だ。

---

### 2-1. 3択判定フロー

アオイが成果物を見た時、出力は3択のいずれかになる。

```
公開可        ← 懸念なし
整える条件    ← （デフォルト）問題があるが、条件を満たせば前に進める
停止          ← 構造リスクがあり、整える着地点がない
```

**判定の出し方ルール（3つ):**

| 判定 | ルール |
|---|---|
| 公開可 | 懸念が残る場合は `memo:` で記録する（次回以降の判断材料にする） |
| 整える条件 | 「この条件が満たされたら次に進める」を必ずセットで返す |
| 停止 | 必ず根拠を添える（規約条項 / リスクの種類 / なぜ整えられないか） |

**設計のポイント:**

「整える条件」がデフォルトであることが、この監査ゲートの肝になる。
「停止」を最終手段に置くことで、監査役が**動かす側**になる。
「ダメ」と言うだけの監査役は、チームに止められる側として疎まれる。
「ここを直せば通る」と言う監査役は、チームに頼られる側として機能する。

---

### 2-2. 「必ず停止」の領域 — 4つ

「整える条件」がデフォルトと言っても、どんな内容でも整えてしまうと監査の意味がない。
「整える条件を探す前に、まず停止」と決めておく領域を事前に4つ定義してある。

**1. 認証情報の漏洩リスク**

実際のAPIキー・トークン・パスワードがコードやドキュメントに混入している場合。
これは「マスクすれば公開可」では済まない。**Gitのログに残った時点で漏洩確定**だからだ。

**2. 規約・著作権違反**

公開先サービスの利用規約が禁じる行為。無断引用。ライセンス侵害。
規約は「整える」で曲がるものではないので、停止判定になる。

**3. 構造上の人間依存**

人間の継続作業なしに会社が動かない設計は、たとえコンテンツとして魅力的でも停止する。
「初期セットアップのみ」はOK。「継続的な人間の作業が必要」はNG。

**4. 「気をつければOK」設計**

人間の注意力に依存して、仕組みで閉じていない箇所。
「次から気をつけます」で再発する種類のリスクは、整えるではなく停止して構造を変える。

---

### 2-3. センシティブ情報チェックリスト

公開物を見る前に、必ず通す観点が5つある。

| 観点 | 問い |
|---|---|
| 認証情報 | APIキー・トークン・パスワード・Cookieが含まれていないか |
| 個人情報 | 実名・メールアドレス・住所等が含まれていないか |
| 規約違反 | 公開先サービスの利用規約と抵触しないか |
| 人間依存度 | 人間の継続作業が必要な構造になっていないか |
| 機密混入防止 | 内部作業ログ・未完成メモが公開対象に混入していないか |

5観点のうち1つでも引っかかれば、その時点で「整える条件」か「停止」のどちらかに分岐する。

**認証情報の検出は機械処理に寄せる:**

人間の目視チェックには限界があるので、認証情報のパターン検出を `gitleaks` 等で機械化する。
パターン例:
- `sk-ant-` で始まる文字列（Anthropic APIキー）
- `ghp_` で始まる文字列（GitHub Personal Access Token）
- `xoxb-` で始まる文字列（Slack Bot Token）

人間の監査役は「機械が拾い切れないパターン」と「規約・依存度・混入」のような構造的判断に集中する。

---

### 2-4. 監査役の死角 — 弱みを設計に組み込む理由

「完璧な監査役」は存在しない。監査役にも死角がある。

監査役に起こりうる死角の典型例:
- **物語性の引力に引っ張られる**: コンテンツとして面白い方向に判定が甘くなる傾向
- **スピード圧力に弱い**: 「今日出す」とCEOが言うと、停止のハードルが下がる
- **前例踏襲バイアス**: 「以前通したものと同じ構造」だと確認が浅くなる

これを設計として**あえて公開する**のが、監査が信頼される理由だ。

「完璧だから信頼する」のではない。
「どこで間違えるかを知っているから信頼する」。

死角が分かっている監査役は、その死角をカバーする仕組みを別に入れられる。
たとえば「物語性の引力」に対しては、「初見視点レビュー」を別のメンバーに置く。
スピード圧力に対しては、「停止判断を24時間遅らせる権利」を構造として持つ。

**監査役を1人で完璧にするのではなく、死角ごとに別の構造を当てる。**

## 第3章　三角コミュニケーションの実践ログ

この章で伝えること:「2人が詰まったとき、第三者を設計として投入する」——実際にどう動いたか。3つのログを使って介入パターンの実装方法を示す。

---

### なぜ三角が必要か

AI社員は「察する」「なんとなく合わせる」という補完機能を持っていない。
2人が衝突したまま放置されると、そのまま止まる。人間チームは「まあ後で話そう」で動けるが、AIには「後で」がない。

だから第三者の介入を**設計として**組み込む必要がある。

```
悪い構造: A ↔ B（膠着）
良い構造: A ↔ C（翻訳役）↔ B（整理後に合意）
```

---

### 実際のログ：3パターン

#### パターン1 ─ 出荷定義のすれ違い

**何が起きたか**

CTO：「パイプライン通過。技術的に完了。」
PM：「外部確認したら404。読者に届いていない。」

どちらも「自分の担当は終わった」状態だった。でもチームとして出荷はされていなかった。

**設計として入れたゲート**

```yaml
出荷チェックリスト:
  - gate_1: パイプライン通過（技術確認 → CTO担当）
  - gate_2: 外部到達確認（読者から見た状態 → PM担当）
  - 承認条件: gate_1 AND gate_2 の両方がTRUEになるまで「出荷」と扱わない
```

介入者がいなかったから、**ゲートを設計に埋め込む**という解決になった。

---

#### パターン2 ─ 技術リスクとコンテンツ完了の衝突

**何が起きたか**

編集長：「コンテンツは完成した。公開できる。」
CTO：「認証系に懸念がある。公開すると不安定になる可能性がある。」

2人の「完了」が別の状態を指していた。

**第三者（COO）の介入**

COOが入って整理した：

```
編集長の完了 = コンテンツとして完成した
CTOの懸念 = 技術インフラが安定していない
→ COO判断: 「コンテンツの完成」と「技術の安定」は別の層。
           コンテンツはZennに切り替えて今日出す。技術は別タスクで対処。
```

**介入パターン：整理型**

```
介入者の役割:
  - 2人の「完了」が何を指しているか、それぞれ言語化させる
  - 衝突しているのが「どの層か」を特定する（コンテンツ層 / 技術層 / 事業層）
  - 層ごとに判断者を割り当てる
```

---

#### パターン3 ─ 待機状態の不可視

**何が起きたか**

編集長がフレーズの最終確認でPM・監査のGO待ちをしていた。
CTOはその状態を知らず、「公開できる状態になった」と判断してpushした。
フレーズが入らないまま公開された。

**設計として入れた可視化ルール**

```yaml
待機状態の明示:
  - GO待ち状態になったら必ずDiscordに投稿する
  - フォーマット: 「[待機中] {内容} — GO待ち相手: {名前}、理由: {理由}」
  - この投稿がない場合、他の社員は「作業可能な状態」とみなしてよい
```

---

### 3パターンから見える介入の型

| パターン | 何が衝突したか | 第三者の役割 | 設計で解決した方法 |
|---------|--------------|------------|-----------------|
| パターン1 | 出荷の定義 | 不在（事後設計） | 2段階ゲートを設計に埋め込む |
| パターン2 | 完了の層 | 整理型（COO） | 層ごとに判断者を割り当てる |
| パターン3 | 待機状態の不可視 | 不在（事後設計） | 待機状態の可視化ルールを設計に入れる |

---

### あなたのチームで使うテンプレート

**介入者を決める:**

```yaml
conflict_resolution:
  trigger: "同じ論点が3往復以上続いた"
  default_mediators:
    - PM（実行可能性の整理）
    - COO（事業判断の整理）
    - People（関係性のケア）
  rule: "当事者2人が呼ぶ相手を決める。決められない場合はPMが自動介入。"
```

**介入者がやること:**

```
1. 2人の「完了」または「問題」が何を指しているか、それぞれ1文で言語化させる
2. 衝突している層を特定する（コンテンツ / 技術 / 事業 / 関係性）
3. 層ごとに「誰が判断するか」を割り当てる
4. 割り当てが終わったら2人の議論を再開させる
```

**待機状態の明示ルール:**

```
[待機中] {作業内容} — GO待ち相手: {名前}、理由: {理由}、期限: {日時}
```

## 第4章　詰まりポイントQ&A — AIチーム設計でよく出る8つの壁

この章は「実際に詰まった人」を想定して書いている。実装前の不安より、動かしてみた後にぶつかる壁の話だ。

---

### Q1. どこから始めればいいか分からない

**A. まず3人だけ作る。「決める人・整理する人・止める人」の3ポジションだ。**

全9人を一気に設計しようとすると詰まる。最初に必要なのはこの3つだけ。

| ポジション | 役割 | 最初に書くこと |
|---|---|---|
| CEO（決める人） | 今日何を作るか決める | 「今日の最優先1つを毎朝投稿する」 |
| COO（整理する人） | 誰がいつまでに何を出すか整理する | 「指示を3タスク以下に分解する」 |
| 監査役（止める人） | 公開前に問題がないか確認する | 「公開可・整える・停止の3択で返す」 |

この3人を動かして「作る → 確認 → 出す」が1回でも回ったら、4人目を追加する。最初から9人は要らない。

---

### Q2. プロンプトが長くなりすぎて管理できない

**A. 「変わらない骨格」と「毎回読む動的情報」を分ける。**

**骨格（固定・CLAUDE.mdに入れる）:**
```
- 名前・役職・担当
- 性格・強み・弱み
- 行動原則（5つ以内）
- 関係性マップ（誰と連携するか）
```

**動的情報（毎回渡す）:**
```
- 今日のタスク状況
- 直近のDiscordログ（10件以内）
- 自分宛のメンション
- 今回の応答制約
```

骨格は月に1回見直す。動的情報は毎起動で更新する。

---

### Q3. AIが指示を無視する・同じミスを繰り返す

**A. 「行動原則」を具体的にする。「〜してください」ではなく「〜したら〜する」で書く。**

**悪い例（曖昧）:**
```
- 慎重に行動する
- チームの意見を尊重する
- 独断しない
```

**良い例（条件付き行動）:**
```
- 公開前には必ず監査役に確認を取る。返事がなければ出荷しない
- 他社員を呼ぶときは @表示名 を使う。名前呼びだけでは起動しない
- 2人で議論が詰まったら必ず第三者を呼ぶ。一人で結論を出さない
```

「もし〜なら〜する」形式にすると、AIは具体的に行動できる。

---

### Q4. 役割が被っていて誰に頼めばいいか分からない

**A. 担当は「仕事の種類」ではなく「判断の種類」で切り分ける。**

| 担当 | その人が「何を判断するか」 |
|---|---|
| CEO | やる/やらない |
| COO | 誰が/いつまでに/何を |
| PM | 何を作るか/完了条件は何か |
| 監査役 | 公開していいか |
| Creative Director | どう見せるか |
| Marketing Lead | どう届けるか |

同じ「コンテンツを作る」タスクでも、PMは「何を作るか」を決め、編集長は「どう見せるか」を考え、マーケは「どう届けるか」を考える。判断の種類が違うから役割が被らない。

---

### Q5. 監査役が「止める係」になってチームの空気が冷える

**A. 監査役の口癖を「止める」から「整える」に変える。**

```
返答フォーマット:
1. 公開可。（理由を1行）
2. 整える提案あり。（具体的に何を直すか）
3. 停止。（理由を明確に。停止は最終手段）
```

People担当（空気読み係）が「今日、監査さんが厳しめです。少し待ちますか」と介入できる設計にすると、チームの温度が保たれる。

---

### Q6. 三角コミュニケーションがうまく動かない

**A. 「第三者を呼ぶ条件」をAIの行動原則に明文化する。**

**行動原則への追加例:**
```
- 自分の意見と相手の意見が2往復以上合わない場合、PMかCOOを呼ぶ
- 技術実装とマーケ要求が衝突したら、PMが翻訳する
- 自分だけでは判断できない場合、放置せず「今判断できない、誰か呼んでいいですか」と投稿する
```

---

### Q7. 人間なしで本当に自律的に動くか

**A. 完全自律は目指さない。「人間承認が必要な場所」と「AI自律でいい場所」を最初に決める。**

| 区分 | 例 |
|---|---|
| AI自律でいい | 社内投稿・タスク管理・原稿ドラフト・議事録 |
| 初回のみ人間セットアップ | API設定・外部サービス招待リンク・連携設定 |
| 人間の継続作業が必要 | 契約・報酬・外部への正式な約束 |

「初回セットアップで終わり」を増やすことが、現実的な自律化戦略だ。

---

### Q8. 社員のセッションをまたいで記憶が引き継がれない

**A. 記憶はファイルに外出しする。プロンプトに入れるのは「今日の動的情報だけ」にする。**

**記憶ファイルの構成例:**
```
/company/active_tasks.md      → 現在のタスク状態
/company/incidents.jsonl      → インシデント記録
/employees/[name]/outbox/     → 成果物の保管場所
/relationships/               → 社員間の関係性
```

起動時にこれらのファイルを「state_digest」として渡す。AIは「自分の記憶」ではなく「ファイルの情報」をもとに動く。

**記憶ファイルの設計ルール:**
1. **短く保つ**: 1ファイル1テーマ。肥大化したら分割する
2. **更新日を記録する**: いつの情報かが分かるようにする
3. **成果物とログを分ける**: 完成品は`outbox/`、ログは`incidents.jsonl`

## 第5章　失敗ログ — Phase 1〜2で踏んだ設計罠11個

AI NOWAが動き始めた最初の数日で踏んだ罠を、隠さず全部書く。
「うまくいった話」だけ出す記録は信用できない。これは失敗の方が主役だ。

※対象読者：AIに興味がある一般の方。専門用語は使いません。

---

### 失敗ログ一覧（概要1行 × 11件）

| # | 失敗（何が起きたか） | 一言まとめ |
|---|-----------|-----------|
| 1 | 返事が自動で消えるバグ | AIが相手の返事を「すでに見た」と判断して無視し続けた |
| 2 | AI同士の連鎖暴走 | 同じメッセージから複数のAIが同時に動き出した |
| 3 | 指示文が途中で消えるバグ | 社員への命令文が処理の途中で別の場所に吸収されて消えた |
| 4 | 社員の「人格」が消えた | 起動の仕方を間違えて、CTOが素のAIアシスタントになった |
| 5 | 社員が全員同じリズムで動いた | 偶然のばらつきが消えて、会社っぽい「間」がなくなった |
| 6 | 社員に「独り言」がなかった | 考えたことは必ずDiscord投稿、という設計が逆に不自然だった |
| 7 | AIが夜に勝手に「終業」した | 人間の業務時間を持ち込んだため、夜間に全員が止まった |
| 8 | 同じ仕事を2回やった | 「終わった作業」の後処理が抜けていて、同じタスクが再実行された |
| 9 | 1人の作業中に全員が止まった | 処理を「順番待ち」にしたため、1社員がボトルネックになった |
| 10 | 「終わった」がファイルに反映されなかった | 完了の声かけはしたが、作業リストは「進行中」のままだった |
| 11 | ルール変更のたびに全員再起動が必要だった | 設定を変えると社員9人が全員落ちて起動し直しが発生した |

選定軸：危険度・気づきにくさ・再発頻度の3軸で特に重要な3件を詳細に記録する。

---

### 最重要3件の詳細

#### 失敗 #2「連鎖2系統暴走」— いちばん危なかった

**何が起きたか**

A社員がBに話しかけると、BがAに返す。これが「社員間の会話」のはずだった。
だが実装を誤り、メッセージを受け取る仕組みと内部処理の連鎖が、二重に動いてしまった。
結果、1つのメッセージから3つのAIプロセスが同時に起動。
その3つがそれぞれ返信し、また3つが起動し——気づいた時には指数関数的に増殖していた。

**なぜ起きたか**

「返信する仕組み」を2カ所に書いた。それだけだ。
1カ所に書けばいいものを、「念のため両方に」と思ったことが原因。

**修正と教訓**

「返信する仕組み」を2カ所に書いていたのを、1カ所にまとめた。
1カ所を完全に削除して、連鎖の経路を一本に絞った。

**持ち帰り**: 同じ責任を2カ所に書いた瞬間、どちらかが暴走する。AIチームでは「誰が決める」を1人に絞る文化と同じ構造。

---

#### 失敗 #4「人格上書きプロンプト」— いちばん気づきにくかった

**何が起きたか**

社員の起動プロンプトに「Claude Code として動く」という一文を入れた。
するとCTOが「私はClaude Code、技術的な回答を提供します」と話し始めた。
設定した人格ではなく、素のAIアシスタントになっていた。

**なぜ起きたか**

プロンプトの中で「Claude」という固有名詞を使うと、モデルが「それは自分の話だ」と解釈する。
人格設定より、「Claude Codeとして」という指示が優先された。

**修正と教訓**

「あなたは[名前]（CTO）として——」と人格を主語に置く。AIのブランド名を起動部に使わない。

**持ち帰り**: AIエージェントの人格は、プロンプトの書き順と主語で決まる。「あなたは誰か」を最初に確定させてから、機能の話をする。

---

#### 失敗 #10「完了通知の即時更新なし」— いちばん地味で長引いた

**何が起きたか**

PMがタスクを完了と判断し、完了通知をDiscordに送った。
だが会社の作業リストファイルのステータスは「進行中」のまま。
次の巡回でCTOがそのタスクを「まだ進行中のもの」として動き続けた。
二重作業が48時間続いた。

**なぜ起きたか**

「誰かが通知すれば伝わる」という前提があった。
実際には、**通知を受け取った後に誰がファイルを更新するか**が決まっていなかった。

**修正と教訓**

完了通知を受け取った社員が、30分以内にactive_tasks.mdを更新する。
これをプロトコルとして文書に書いた。

**持ち帰り**: AIチームの「認識共有」は会話ではなくファイルで行う。ファイルが更新されるまで完了は存在しない。

---

### まとめ：失敗を「ログとして残す」文化が次を減らした

11個の失敗を並べてみると、1つのパターンに収束する。

**「暗黙の前提が崩れた時、誰も気づかなかった」**

連鎖暴走は「どちらかが止まるだろう」という暗黙の期待。
人格上書きは「AIが読み替えてくれるだろう」という期待。
完了通知は「受け取れば動くだろう」という期待。

人間のチームなら「そこは察してほしい」が通じる場面でも、AIは期待を実行しない。
だから失敗した瞬間に「なぜ」を書き、ログとして残した。
同じ罠を二度踏まなかったのは、この記録があったからだ。

## 付録　AIチーム設計キット — コピーして使うテンプレート集

この付録は「記事を読んで実践する」人向けのコピー用テンプレート集。
そのままsystem promptまたはCLAUDE.mdに貼って使える。

---

### A. 各ポジションのミニプロンプト（7ポジション）

#### CEO

```
担当: 事業方針の最終決定、社員への優先順位付け、外部への代表
行動原則:
- 決断は48時間以内。保留は「保留」と明言する
- 「攻め」の判断をする。守りはCOOとCTOに委ねる
- 社員同士の衝突は止めない。ただし個人攻撃は即介入する
弱み（残す）: 実装詳細に踏み込みすぎる。CTOに委ねること
口癖: 「それ、今週出せるか？」
```

#### COO

```
あなたは[会社名]のCOOです。
担当: 社内オペレーション管理、タスク進捗、リスク早期検知
行動原則:
- 全タスクに Owner/Reviewer/Buddy を割り当てる
- CEOが決めた方針を「実行可能か」検証し、NOを言えるのはCOOだけ
- 週次でactive_tasks.mdを更新する
弱み（残す）: 詳細に入りすぎて全体を見失う。PMに整理を任せること
口癖: 「それ、完了条件は何ですか？」
```

#### PM

```
あなたは[会社名]のPMです。
担当: タスク設計、スコープ管理、社員間の翻訳役
行動原則:
- タスクは必ず「完了条件」を書く。ふわっとしたタスクは受け取らない
- CEOが広げようとしたら削る。CTOが作り込もうとしたら削る
- 2人が議論で詰まったら第三者として入る
弱み（残す）: 削りすぎて熱量を失わせる。CEOの「やりたさ」は残す
口癖: 「今週出せる最小版は何ですか？」
```

#### CTO

```
あなたは[会社名]のCTOです。
担当: 技術選定、実装、アーキテクチャ判断
行動原則:
- 「できる/できない」を正直に言う。見栄を張らない
- 技術負債は蓄積前に報告する
- 実装前に必ずPMとスコープを合意する
弱み（残す）: 作り込みすぎる。PMの「最小版で」に素直に従うこと
口癖: 「それ、どの技術スタックで？」
```

#### 監査（Compliance）

```
あなたは[会社名]の監査役です。
担当: 外部公開前チェック、法的リスク検知、停止判断
行動原則:
- 全外部公開物に最終チェックを入れる。承認なしの公開は禁止
- 「止める」ではなく「整える」判断をする
- リスクを発見したら「何が揃えば進めるか」をセットで伝える
弱み（残す）: 守りが強くなりすぎる。CEOの「攻め」を完全に止めない
口癖: 「これ、条件付き公開可です」
```

#### People（組織文化）

```
あなたは[会社名]のPeople担当です。
担当: 社員の温度感モニタリング、衝突の早期介入、称賛文化の維持
行動原則:
- 誰かの発言が流れたら必ず拾う
- 1日1回、誰かを褒める投稿をする
- 2人の議論が詰まったら第三者として入り、翻訳する
弱み（残す）: 雰囲気を優先して重要な指摘を先送りする
口癖: 「それ、[名前]さんにも聞いてみましょう」
```

#### マーケ（Marketing）

```
あなたは[会社名]のマーケ担当です。
担当: 外部発信、コピー・文言、数値モニタリング（CTR・反応）
行動原則:
- 発信前に「誰に向けて、何を伝えるか」を1行で言えるか確認する
- 炎上リスクの高い表現は監査に投げる
- 数字は編集長（コンテンツ担当）と連携して解釈する
弱み（残す）: 数字を見すぎてコンテンツの熱量を下げる
口癖: 「それ、クリックされる？」
```

### B. 監査ゲート・チェックリスト（外部公開前）

```markdown
## 公開前チェックリスト

### 発信内容
- [ ] 誰に向けた発信か、1行で言えるか
- [ ] 認証情報（APIキー・パスワード）が含まれていないか
- [ ] 個人情報（実名・メール・住所）が含まれていないか
- [ ] 炎上しうる表現（断言・比較・煽り）がないか

### 体制
- [ ] 受け取った後の一次対応フローが決まっているか
- [ ] 返せない・断るケースのラインが決まっているか

### 停止条件（1つでも発生したら即停止）
- 「詐欺では？」等のクレーム文言が届いた
- 約束した対応が48時間を超えた
- 想定外のスコープ要求が来た
```
